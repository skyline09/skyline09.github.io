编写可读代码的艺术

1. 代码应当易于理解

好代码不是越长也好（可能存在冗余、拖沓）也不是越短越好（可能存在过于简写、hack逻辑），而是尽量减短 别人阅读并理解代码的时间，降低后期维护成本。

2. 把信息装到名字里

  • 选择专业的词
    不要滥用get、set，这两个名字一般用于class的private变量的存取函数
    get –> fetch, extract, download, compute
    size –> height, numNodes, memoryBytes
    send –> deliver, dispatch, announce, distribute, route
    find –> search, extract, locate, recover
    start –> launch, create, begin, open
    make –> create, set up, build, generate, compose, add, new

  • 尽量避免tmp、ret这样泛泛无实义的名字
    tmp名字有一个很经典的应用场景：对象的持久化之前创建一个snapshot

  • 循环迭代器
    当函数内有多次使用循环迭代器时，不要仅仅使用 i，j，it，iter这样简单的变量，而是要加上循环变量简称加以区分 (Idx表示index，it表示iterator)
    例如：packageInfos迭代时–> pIdx, pIt, dataInfos迭代时–> dIdx, dIt

  • 变量带单位信息
    | 函数参数 | 带单位的参数 |
    | —————- | ————————— |
    | start(int delay) | delay –> delay_secs |
    | createCache(int size) | size –> size_mb |
    | ThrottleDownload(float limit) | limit –> max_kbps |
    | Rotate(float angle) | angle –> degrees_cw |

  • 名字长度
    变量名字应该充分表达变量信息，作用域越小的变量，名字可以适当简短，而全局变量则不行

  • 单词缩略
    变量名字应该充分表达变量信息，作用域越小的变量，名字可以适当简短（采用缩略单词）
    applicatonMasterManager –> appMasterManager
    evaluation –> eval
    string –> str

  • 使用大小写、下划线附带信息
    class成员变量一律使用 _processInfos风格（下划线开头），非成员变量一律使用 proessInfos风格（无下划线）

3. 不会误解的名字

  • 避免使用二义性单词

此处filter容易造成误解，是 “挑出“ 还是 “去掉“ ？

  • 表示包含、极限语义的时候，不要使用limit（二义性），要用 max、min这些
  • 使用 [first, last] 表示 全包含的范围
  • 使用 [begin, end) 表示 半包含的范围
  • 布尔语义的名词 加上 is, has, can, should, disable, enable, use, need 这样的前缀

4. 审美

  • 杜绝重复代码，一定要思考如何抽象避免重复代码
  • 代码逻辑分成 段落，段落之间放置一个空行

5. 注释

  • 不要给不好的名字加注释 – 应该把名字改好， 好代码 > 坏代码 + 好注释
  • 好注释一般是 指导性批注、代码中的缺陷（TODO）、需要用户注意的意料之外行为

6. 注释的写法

  • 避免使用不明确的二义性，将二义性 用名词具体化
    // Insert the data into the cache, but check if it’s too big first. shit, check who ?
    // Insert the data into the cache, but check if the data too big first. OK

7. 将控制流变得易读

  • if/else 处理 正逻辑、负逻辑的顺序 视情况而定，分布式系统中一般是 code for failure
  • 避免do/while 循环
  • 注意 布尔运算时的 短路逻辑

8. 拆分超长的表达式

  • 使用总结性变量来代替一大块代码

改写之后：

  • 反向思维 简化逻辑

  • 适当使用宏函数来简化重复代码

使用宏函数改写之后：

9. 变量与可读性

  • 变量越多，越难全部追踪他们的动向
  • 变量在初次使用时定义，而不是C语言中将所有变量定义放在函数或语句块顶端

10. 抽取不相关的子问题

  • 将通用型代码抽取出来，封装成util类、common模块
  • 函数中 有一些代码是为函数设置输入而进行的前期准备工作，或者对函数输出做后期处理的，这些代码与函数主要逻辑是不相关的，并且会影响到 别人阅读代码，这些代码应该抽取到独立函数。

11. 一次只做一件事

  • 分析代码逻辑，将多个任务拆分到每一个小函数里面

12. 把想法变成代码

13. 少写代码

  • 程序员经常低估实现一个功能所花的时间

  • 经常阅读、熟悉、复用标准库函数，不要自己去造蹩脚的轮子

  • 用于删除无用代码
